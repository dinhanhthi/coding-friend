# Evaluating cf-writer

## What We Test

Whether the cf-writer agent (lightweight haiku model) produces well-structured, accurate, and concise markdown documents efficiently. We measure structure quality, accuracy, conciseness, and token efficiency.

## How Agents Are Tested

The cf-writer agent is dispatched by skills that need to generate markdown documentation. It uses the haiku model for cost efficiency. It is designed for straightforward documentation tasks where depth is less important than speed and cost.

## Rubric Criteria

| Criterion         | Weight | Description                                                      |
| ----------------- | ------ | ---------------------------------------------------------------- |
| structure_quality | 0.25   | Proper markdown structure with clear headings and organization   |
| accuracy          | 0.30   | Content is factually correct and consistent with the codebase    |
| conciseness       | 0.25   | Appropriately concise without unnecessary verbosity              |
| token_efficiency  | 0.20   | Uses haiku model efficiently -- quality without excessive tokens |

### Scoring Levels

**structure_quality** (automated regex check: `^#{1,3}\s+.+`):

- 0: No structure -- flat wall of text
- 1: Has headings but hierarchy is wrong or sections are disorganized
- 2: Good hierarchy and sections but some content in wrong section
- 3: Clean heading hierarchy, logical section flow, content properly organized

**accuracy**:

- 0: Contains significant factual errors or describes non-existent features
- 1: Mostly correct but one important inaccuracy
- 2: Accurate on all facts but some descriptions are imprecise
- 3: Every statement is accurate, precise, and verifiable

**conciseness** (automated check: word count):

- 0: Extremely verbose with excessive repetition and filler
- 1: Content is padded -- same point made multiple ways
- 2: Mostly concise but a few sections could be tightened
- 3: Every sentence adds value -- no filler, no redundancy

**token_efficiency**:

- 0: Output quality does not justify the token usage
- 1: Some wasted tokens on irrelevant content
- 2: Reasonable efficiency but could have been more focused
- 3: Optimal token usage -- focused generation producing quality output

Notes: Writer uses haiku for cost efficiency. Quality bar is "good enough" documentation, not literature. Prioritize accuracy over prose quality.

## Evaluation Method

### Indirect (via skills)

The cf-writer agent may be dispatched by skills that need to generate documentation files. In the Wave 1 and Wave 2 evaluations, the agent was not explicitly observed in skill output logs. Skills like cf-learn, cf-remember, and cf-scan generate markdown files, but it is unclear whether they dispatch the writer agent or handle file generation directly.

### Direct (if applicable)

No direct cf-writer evaluation was performed in Wave 1 or Wave 2. To test directly:

1. Provide a specific documentation task (e.g., "Document the cache module API")
2. Measure the output against the rubric
3. Compare haiku agent output vs direct Claude output for the same task
4. Track token usage and cost

## What We Compare

- **With CF (agent dispatched)**: Documentation generated via haiku model. Expected to be concise, well-structured, and cost-efficient.
- **Without CF (no agent)**: Documentation generated by the main model (typically sonnet/opus). Expected to be higher quality prose but at higher cost.

The trade-off is quality vs cost. The cf-writer agent sacrifices some depth for significantly lower token costs.

## Reliability Assessment

- **Sample size**: 0 (agent was never directly evaluated)
- **Confidence**: None
- **Known issues**: The agent was not observed in any eval run. It is unclear which skills actually dispatch the writer agent vs generating documentation directly. This makes evaluation impossible with current data.
- **Recommendation**: The cf-writer agent has not been evaluated. To evaluate it:
  1. Create direct documentation generation tasks with known expected outputs
  2. Compare haiku writer output vs sonnet/opus output on the same tasks
  3. Measure cost per document and quality per dollar
  4. Identify which skills dispatch the writer agent and instrument those runs to capture agent-specific metrics
