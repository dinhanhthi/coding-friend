## Tasks

1. **7.1 MCP server registration**
   - Files: `plugin-codex/.mcp.json` (new, generated by build script in Phase 3.3 extended here), `cli/src/commands/init.ts` (extended in 4.5 — verify it writes `[mcp_servers.coding-friend-memory]` to `~/.codex/config.toml`)
   - `plugin-codex/.mcp.json` content:
     ```json
     {
       "mcpServers": {
         "coding-friend-memory": {
           "command": "npx",
           "args": ["-y", "coding-friend-cli", "mcp-serve", "docs/memory"],
           "env": {}
         }
       }
     }
     ```
   - The `mcp-serve <memoryDir>` command starts the memory MCP server over stdio. `cf mcp` remains the setup/health helper that writes `.mcp.json` for Claude-compatible clients. Same server binary serves both hosts.
   - Verify: `codex` interactive session lists the memory tools (`memory_search`, etc.) alongside built-ins.
   - Rollback: remove `.mcp.json` entry; remove TOML lines.

2. **7.2 Cohabitation smoke test**
   - Files: `cli/src/lib/__tests__/memory-cohabitation.test.ts` (new — e2e-style with skip-on-CI flag)
   - Scenario:
     - Open one Claude session (mock), call `memory_store` to insert a memory
     - Open one Codex session (mock), call `memory_search` for that memory — must return it
     - Reverse: Codex stores, Claude reads
     - Concurrent: two writes in quick succession — SQLite WAL must handle without lock errors
   - Verify: all 3 paths pass; the test runs by default but is skipped in CI environments without both binaries (gated by `commandExists`).
   - Rollback: delete test.

## Exit criteria

- `codex` can call MCP memory tools after `cf install --codex`
- Memory data written from Claude session is visible in a Codex session and vice versa
- Concurrent-write test passes
