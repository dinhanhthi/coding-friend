The memory system gives Claude Code persistent project memory across sessions. It stores facts, preferences, decisions, debug episodes, and procedures as markdown files in `docs/memory/`, with optional indexing for fast hybrid search.

## How It Works

There are two ways to capture memories:

1. **Manually** — use the [`/cf-remember`](/docs/skills/cf-remember/) skill during a session. Tell Claude what to remember and it writes a markdown file to `docs/memory/`, categorized and indexed automatically.

2. **Automatically** — when `memory.autoCapture` is enabled in `.coding-friend/config.json`, a PreCompact hook fires before Claude Code compacts the session context. It captures a summary of the current session (main task, key decisions, findings) as an [`episode`](#memory-types) memory.

Other skills also auto-capture when they find notable insights:

- [`/cf-fix`](/docs/skills/cf-fix/) captures bug root causes
- [`/cf-review`](/docs/skills/cf-review/) captures architectural patterns
- [`/cf-sys-debug`](/docs/skills/cf-sys-debug/) captures debugging breakthroughs

All captured memories are plain markdown files — human-readable, git-trackable, and never locked in.

### [`/cf-remember`](/docs/skills/cf-remember/) vs [`cf memory`](/docs/cli/cf-memory/)

- **`/cf-remember`** is a skill you use _inside a Claude Code session_. Say `/cf-remember` and describe what to save. Claude analyzes the conversation, picks the right category, and writes a markdown file with proper frontmatter. It also calls the `memory_store` MCP tool to index the memory for fast search.

- **`cf memory`** is a CLI command you run _in your terminal_. It manages the memory system itself.

### How Search Works

When Claude searches your memories (via `memory_search`), the system routes the query through the best available backend:

#### Step 1: Query Routing

The query is analyzed to pick the best search mode:

- **Quoted strings** or code patterns (e.g., `"auth middleware"`, `config.db.host`) — **keyword search** (FTS5 BM25)
- **Questions** starting with who/what/why/how/when/where — **semantic search** (vector similarity)
- **Everything else** — **hybrid search** (both keyword + semantic, fused with RRF)

#### Step 2: Keyword Search (FTS5)

FTS5 performs full-text search with BM25 ranking. Fields are weighted:

| Field       | Weight | Why                                               |
| ----------- | ------ | ------------------------------------------------- |
| title       | 10x    | Most specific — exact match here is strong signal |
| tags        | 6x     | Curated keywords the user chose                   |
| description | 4x     | One-line summary                                  |
| content     | 1x     | Full body text (broad match)                      |

#### Step 3: Semantic Search (Embeddings)

For semantic search, the system converts both the query and each stored memory into **embedding vectors** — high-dimensional numeric representations that capture meaning, not just keywords.

**How embeddings work:**

1. When you **store** a memory, the system combines its title + description + tags + content into a single text block
2. This text is passed to an **embedding model** (see [Embedding Models](#embedding-models) below) which outputs a vector of N dimensions (e.g., 384 floats for the default model)
3. The vector is stored in a **sqlite-vec** table alongside the memory ID
4. Embeddings are **cached by content hash** — unchanged content is never re-embedded

When you **search**, the query text goes through the same embedding model to produce a query vector. Then **cosine similarity** finds the stored vectors closest to the query — meaning semantically similar content is returned even if no keywords match.

#### Step 4: Fusion (RRF)

In hybrid mode, both keyword and semantic results are merged using **Reciprocal Rank Fusion** (RRF, k=60). Each result gets a score based on its rank in each list, and the final ranking combines both signals.

#### Step 5: Post-processing

- **Temporal decay** — Recent memories score higher. A 90-day half-life retains 70% of the base score.
- **Type/tag filtering** — Optional filters narrow results before returning.

## Architecture

```
Claude Code Session
        |
+-------+--------+
|  MCP Server     | <- stdio transport
|  (thin client)  |
+-------+--------+
        |
        v
+------------------------------------+
|     Memory Daemon                  |
|     (Hono + Unix Domain Socket)    |
|                                    |
|  Tier 1: SQLite + FTS5 + vec      |
|  Tier 2: MiniSearch (in-memory)   |
|  Tier 3: Markdown (file grep)     |
+------------------------------------+
        |
        v
+------------------+
|  docs/memory/    |
|  (source of truth)|
+------------------+
```

### MCP Integration

The MCP (Model Context Protocol) server is the bridge between Claude Code and the memory backends. It runs as a [stdio transport](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports#stdio) — Claude Code spawns it automatically when it needs memory tools. You don't need to start it manually.

The MCP server exposes 6 tools that Claude Code calls during sessions:

| Tool              | Purpose                                   |
| ----------------- | ----------------------------------------- |
| `memory_store`    | Store a new memory                        |
| `memory_search`   | Search memories (keyword/semantic/hybrid) |
| `memory_retrieve` | Get a specific memory by ID               |
| `memory_list`     | List memories with filtering              |
| `memory_update`   | Update existing memory                    |
| `memory_delete`   | Delete a memory                           |

When you use `/cf-remember`, Claude calls `memory_store` behind the scenes. When a skill needs context from past sessions, it calls `memory_search`. You never interact with these tools directly — they're part of the automatic workflow.

To configure the MCP server for your project, run `cf memory mcp` to get the configuration snippet for your Claude Code settings.

### 3-Tier Graceful Degradation

The system automatically detects and uses the best available backend:

- **Tier 1: SQLite** — Full-power hybrid search with [FTS5 full-text](https://www.sqlite.org/fts5.html) + [sqlite-vec](https://github.com/asg017/sqlite-vec) semantic vectors, fused with Reciprocal Rank Fusion (RRF). Requires `cf memory init` to install native dependencies.
- **Tier 2: MiniSearch** — In-memory BM25 index with fuzzy matching. Activates via the daemon (`cf memory start`) when SQLite deps aren't installed.
- **Tier 3: Markdown** — Zero-dependency fallback. For each file, it checks frontmatter (title, description, tags) first — if that matches, it stops there. It only reads the full file content for files whose frontmatter didn't match. Always available, no setup needed.

Tier detection happens automatically:

```
SQLite deps available?  → Tier 1
Daemon running?         → Tier 2
Otherwise               → Tier 3
```

## Memory Types

| Type         | Category\*        | Question it answers             | Example                                     |
| ------------ | ----------------- | ------------------------------- | ------------------------------------------- |
| `fact`       | `features/`       | "How does this work?"           | "Auth module uses JWT in httpOnly cookies"  |
| `preference` | `conventions/`    | "How does the user want it?"    | "Always use async/await, never .then()"     |
| `context`    | `decisions/`      | "What state is the project in?" | "Migrating from REST to GraphQL"            |
| `episode`    | `bugs/`           | "What did we do before?"        | "Debugged CORS on /api/upload"              |
| `procedure`  | `infrastructure/` | "What steps to do X?"           | "Deploy: build -> ship -> merge -> release" |

\*Category is the subfolder inside `docs/memory/` where the markdown file is stored. Each type maps to a folder so memories are organized on disk and can be browsed or edited manually.

## Memory Importance

Importance is a integer from 1 to 5 (default: 3). It is used for filtering and future ranking features.

- 1: Low importance
- 2: Medium importance
- 3: High importance
- 4: Very high importance
- 5: Critical importance

Importance is set automatically for auto-captured sessions (episodes) to 2, and manually saved memories default to 3. It can be overridden by setting the `importance` field in the frontmatter. You can also use `/cf-remember` to set the importance to a specific value. For example,

```bash
/cf-remember this is critical, importance 5
```

## Markdown Format

Every memory is a markdown file with YAML frontmatter:

```yaml
---
title: "API Authentication Pattern"
description: "Auth module uses JWT tokens stored in httpOnly cookies with RS256"
type: fact
tags: [auth, jwt, api, security]
importance: 3
created: 2026-03-12
updated: 2026-03-12
source: conversation
---
The auth module uses RS256-signed JWT tokens...
```

### Fields

- **`title`** — Short, descriptive name
- **`description`** — One-line searchable summary (under 100 chars, grep-friendly). This is critical for search quality — be specific. Good: `"Config management via .coding-friend/config.json with local/global override"`. Bad: `"How we manage config"`.
- **`type`** — One of `fact`, `preference`, `context`, `episode`, `procedure`
- **`tags`** — Keywords for filtering and search ranking
- **`importance`** — Integer from 1 to 5 (default: 3). Indicates how critical this memory is.
- **`created`** / **`updated`** — ISO date strings
- **`source`** — Where this memory came from: `conversation` (manual via `/cf-remember`) or `auto-capture` (automatic via PreCompact hook)

## Search Modes

The system automatically routes queries based on the input:

- **Quoted strings** or code patterns — keyword search (FTS5)
- **Questions** (who/what/why/how) — semantic search (vector similarity)
- **Default** — hybrid search (FTS5 + vector, fused with RRF)

Results are ranked with:

- **BM25 scoring** with weighted columns: title (10x), tags (6x), description (4x), content (1x)
- **Temporal decay** — recent memories score higher (90-day half-life, retaining 70% base score)
- **Deduplication** — warns when storing near-duplicate content (Jaccard similarity)

## Embedding Models

Semantic search in Tier 1 requires an **embedding model** to convert text into vectors. CF Memory supports two providers:

### Providers

**Transformers.js** (default) — Runs locally in the Node.js process. No external service needed. The model is downloaded once (~23 MB) and cached in `~/.coding-friend/memory/node_modules/`.

**Ollama** (optional) — Uses a locally running [Ollama](https://ollama.com/) server. Faster for repeated queries and supports a wider range of models. The embedding model runs on the Ollama server, not in the Node.js process.

### Default Model

Both providers default to **all-MiniLM-L6-v2** (384 dimensions) — a compact model optimized for short text similarity. This covers most use cases well.

### Recommended Models

| Model                      | Dims | Size    | Provider        | Notes                                                   |
| -------------------------- | ---- | ------- | --------------- | ------------------------------------------------------- |
| `all-minilm:l6-v2`         | 384  | ~23 MB  | Ollama          | **Default** — fast, good for short text                 |
| `nomic-embed-text`         | 768  | ~274 MB | Ollama          | **Recommended upgrade** — better semantic understanding |
| `mxbai-embed-large`        | 1024 | ~670 MB | Ollama          | Best quality, good for large memory collections         |
| `snowflake-arctic-embed:s` | 384  | ~67 MB  | Ollama          | Alternative small model                                 |
| `bge-base-en-v1.5`         | 768  | ~130 MB | Ollama          | Good general-purpose, English-optimized                 |
| `Xenova/all-MiniLM-L6-v2`  | 384  | ~23 MB  | Transformers.js | Default (auto-downloaded)                               |

### Using Ollama

1. [Install Ollama](https://ollama.com/download)
2. Pull an embedding model:
   ```bash
   ollama pull nomic-embed-text
   ```
3. Configure in `.coding-friend/config.json`:
   ```json
   {
     "memory": {
       "embedding": {
         "provider": "ollama",
         "model": "nomic-embed-text"
       }
     }
   }
   ```
4. Rebuild to re-embed existing memories with the new model:
   ```bash
   cf memory rebuild
   ```

If Ollama is not running or the model is unavailable, the system automatically falls back to Transformers.js.

### Changing Models

When you switch to a model with **different dimensions** (e.g., 384 → 768), existing embeddings become incompatible. The system detects this automatically:

1. On startup, CF Memory checks if the configured model's dimensions match the stored embeddings
2. If mismatched, **vector search is disabled** and a warning is logged
3. Run `cf memory rebuild` to re-embed all memories with the new model
4. After rebuild, vector search is re-enabled with the new dimensions

Markdown files are never affected — they remain the source of truth regardless of embedding changes.

### Configuration

Configure embedding settings in `.coding-friend/config.json`:

```json
{
  "memory": {
    "embedding": {
      "provider": "ollama",
      "model": "nomic-embed-text",
      "ollamaUrl": "http://localhost:11434"
    }
  }
}
```

Or via environment variables (useful for MCP server configuration):

| Variable                      | Default                  | Description                    |
| ----------------------------- | ------------------------ | ------------------------------ |
| `MEMORY_EMBEDDING_PROVIDER`   | `"transformers"`         | `"transformers"` or `"ollama"` |
| `MEMORY_EMBEDDING_MODEL`      | (provider default)       | Model name                     |
| `MEMORY_EMBEDDING_OLLAMA_URL` | `http://localhost:11434` | Ollama server URL              |

See [`config.json` reference](/docs/configuration/config-json#memory) for all memory configuration options.
