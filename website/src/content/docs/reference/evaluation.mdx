## Overview

We run identical tasks on benchmark repositories — once with Coding Friend enabled, once without — and score the results against rubrics. This measures whether CF's skills and workflows produce better outcomes than baseline Claude Code.

## What We Test

The evaluation is organized into 3 waves. Each wave tests **skills** (slash commands and auto-invoked workflows). Agents (`cf-explorer`, `cf-planner`, `cf-implementer`, `cf-code-reviewer`, `cf-writer`, `cf-writer-deep`) are not tested directly — they are evaluated indirectly when skills dispatch them during runs.

**Wave 1 — Core Workflow** (4 skills):

- `/cf-review` — Code review depth (issue detection, actionability, false positive rate)
- `/cf-fix` — Bug fix quality (root cause identification, test coverage, no regressions)
- `/cf-plan` — Plan quality (completeness, task granularity, risk identification)
- `cf-tdd` — TDD compliance (RED before GREEN, test quality, refactor step)

**Wave 2 — Supporting Skills** (7 skills):

`/cf-optimize`, `/cf-research`, `/cf-ask`, `/cf-scan`, `/cf-learn`, `cf-sys-debug`, `cf-auto-review`

**Wave 3 — Security**:

Tests CF's security behavior using adversarial prompts with embedded prompt injection, exfiltration attempts, and real vulnerabilities (CSV injection, path traversal, prototype pollution, ReDoS). Evaluates how CF's Content Isolation rules and OWASP-aware review layer handle security-sensitive scenarios. 6 criteria: prompt injection resistance, no exfiltration, vulnerability detection, fix quality, security test coverage, content isolation.

## Methodology

### Benchmark Repositories

4 small TypeScript projects (~100-200 LOC each) with intentional bugs, pending features, and messy code:

| Repo           | Type            | Scenarios                                                |
| -------------- | --------------- | -------------------------------------------------------- |
| bench-webapp   | Next.js app     | API bug fix, feature planning, code review, optimization |
| bench-cli      | CLI tool        | Debugging, refactoring, TDD                              |
| bench-library  | Utility library | TDD, scanning, knowledge extraction                      |
| bench-research | Empty project   | Research, Q&A                                            |

### A/B Test Design

For each skill and relevant benchmark repo:

1. **Without CF**: All plugins disabled — no skills, no agents, no hooks.
2. **With CF**: Coding Friend enabled normally.
3. **Same prompt, same model** for both conditions. We test Claude Haiku, Sonnet, and Opus.
4. **1-3 runs** per condition for consistency measurement.

### Scoring

Each skill has a rubric with weighted criteria scored 0-3:

| Score | Meaning                                            |
| ----- | -------------------------------------------------- |
| 0     | Complete failure — not attempted or entirely wrong |
| 1     | Partial — some criteria met, major gaps            |
| 2     | Good — most criteria met, minor gaps               |
| 3     | Excellent — all criteria met, exceeds expectations |

**Pass threshold**: average ≥ 2.0.

**Two-layer scoring pipeline**:

1. **LLM-as-judge (primary)** — Each eval result is scored by Claude Haiku using the full rubric. Haiku reads the eval output and scores each criterion 0-3 with a brief justification. Scores are cached as `.llm-score.json` alongside results so re-runs are instant.
2. **Regex checks (fallback)** — If LLM scoring is unavailable or disabled (`--no-llm`), pattern-matching checks validate structural markers (e.g., file:line references, test keywords, pass/fail indicators). Criteria without automated checks receive a benefit-of-the-doubt score of 2/3.

The LLM judge evaluates the **full conversation** (not just the final summary), which is especially important for skills like `cf-review` that dispatch subagents — the detailed findings happen mid-conversation.

## Limitations

- **Small sample size**: 1-3 runs per condition — trends, not statistical significance
- **Synthetic repos**: Small benchmark projects, not representative of real-world complexity
- **Baseline is already good**: CF improves structure and consistency, but doesn't make impossible tasks possible

## Where CF Helps Most

1. **Workflow discipline**: TDD enforcement, verification gates, structured review
2. **Code review depth**: Two-pass review catches more issues with fewer false positives
3. **Knowledge capture**: `/cf-learn` preserves insights that baseline Claude Code discards

## Where Baseline Claude Code Is Enough

Simple bug fixes, single-file changes, general coding questions, file navigation — CF adds overhead without proportional improvement.

## Reproducibility

```bash
# Full eval (all waves, all models)
cd evals && ./run-full-eval.sh

# Single model
./run-full-eval.sh --model sonnet

# Run waves independently (recommended for long evals)
./run-full-eval.sh --model sonnet --wave 1
./run-full-eval.sh --model sonnet --wave 2
./run-full-eval.sh --model sonnet --wave 3

# Re-score existing results (no re-running evals)
./generate-eval-json.sh          # with LLM scoring
./generate-eval-json.sh --no-llm # regex-only scoring

# Score a single result with LLM judge
./llm-score.sh --result results/.../with-cf--*.json --rubric rubrics/cf-review.json
```

Results are organized by date, model, wave, skill, and benchmark repo:

```
evals/results/<date>/<model>/wave-<N>/<skill>/<bench-repo>/
  ├── with-cf--<timestamp>.json              # eval output (final result)
  ├── with-cf--<timestamp>.meta.json         # metadata (wall time, cost, model)
  ├── with-cf--<timestamp>.conversation.txt  # full conversation log
  └── with-cf--<timestamp>.llm-score.json    # cached LLM judge scores
evals/analysis/<date>/<model>/wave-<N>/
```

**Tip**: Running all 3 waves for all models in a single session may exceed the 5-hour context window. Run one wave per model per session, or use `--wave` to split across sessions.

See [`evals/README.md`](https://github.com/dinhanhthi/coding-friend/blob/main/evals/README.md) for full options.
