The `/cf-review` skill performs comprehensive code review using a dedicated subagent. It examines code from multiple angles — security, performance, style, and test coverage — with review depth automatically scaled to the size and sensitivity of your changes.

## Usage

```bash
/cf-review [target]
```

Different behavior is expected based on the value of `[target]`:

- **No arguments** — reviews all uncommitted changes (`git diff` + `git diff --staged`)
- **File or directory** — reviews that specific path
- **Commit range** — reviews a range of commits
- **Natural language description** — reviews all uncommitted changes, focused on the described area

### Flags

- `--deep` / `--quick` — force a review depth instead of auto-detecting
- `--with-codex` / `--codex` — run a [Codex](https://developers.openai.com/codex/cli/) second-opinion review in parallel with Claude's, then merge both into one report (see [Dual Review with Codex](#dual-review-with-codex))
- `--claude` / `--with-claude` — run a headless Claude CLI review in parallel (skipped when already in Claude Code — `HOST=claude`)
- `--gemini` / `--with-gemini` — run a headless [Gemini CLI](https://github.com/google-gemini/gemini-cli) review in parallel
- `--cursor` / `--with-cursor` — run a headless [Cursor Agent](https://cursor.com) review in parallel (`cursor-agent --mode ask`)
- `--grok` / `--with-grok` — run a headless Grok CLI review in parallel
- `--out` — run Claude's review first, then export a [`/cf-review-out`](/docs/skills/cf-review-out/)-style prompt file with Claude's findings embedded for the manual [`/cf-review-in`](/docs/skills/cf-review-in/) round-trip. Mutually exclusive with the auto reviewer flags above.

All headless reviewer flags (`--claude`, `--gemini`, `--cursor`, `--grok`) review the **exact diff** Claude's own review sees (via `gather-diff.sh`). They apply only to the default uncommitted-changes target — not file paths or commit ranges. Each agent runs read-only and degrades gracefully if unavailable, errors, times out, or finds no changes.

## Examples

```bash
# Review all uncommitted changes
/cf-review
```

```bash
# Review a specific directory
/cf-review src/auth/
```

```bash
# Review a single file
/cf-review src/api/handlers/user-controller.ts
```

```bash
# Review the last 3 commits
/cf-review HEAD~3..HEAD
```

```bash
# Review changes related to a specific topic
/cf-review the changes related to the auth logic
```

```bash
# Force a deep or quick review
/cf-review src/auth/ --deep
/cf-review README.md --quick
```

```bash
# Review with Claude AND Codex, merged into one report
/cf-review --with-codex
```

```bash
# Review with Claude AND Gemini + Cursor, merged into one report
/cf-review --gemini --cursor
```

```bash
# Run Claude's review, then export a prompt for manual external review
/cf-review --out
```

## Workflow

1. **Identify target** — Determine what to review based on `[target]` (see [Usage](#usage))
2. **Gather diff** — Collect changes via `git diff` and assess change size
3. **Select review mode** — Auto-detect QUICK, STANDARD, or DEEP based on size and sensitivity
4. **Dispatch to orchestrator** — Dispatch a [`cf-reviewer`](/docs/reference/agents/#cf-reviewer-orchestrator) orchestrator agent via the Agent tool (isolated context — clean, unbiased, token-efficient)
5. **Parallel multi-agent review** — The orchestrator dispatches 5 specialist agents in parallel (plan alignment, security, code quality + slop detection, test coverage, project rules), then merges results via a reducer agent that deduplicates and severity-ranks findings
6. **Run dedicated security scan** — Invoke Claude Code's built-in [`/security-review`](https://github.com/anthropics/claude-code-security-review) for an additional security-focused pass, then merge and deduplicate findings with Layer 3 results
7. **Report findings** — All findings from every layer collected into 4 categories (🚨 Critical / ⚠️ Important / 💡 Suggestion / 📋 Summary), each tagged with source layer

### Review Modes

| Mode         | When                                              | What Happens                                                            |
| ------------ | ------------------------------------------------- | ----------------------------------------------------------------------- |
| **QUICK**    | ≤3 files, ≤50 lines, no sensitive paths           | Secret scan + obvious issues only. Fast.                                |
| **STANDARD** | 4–10 files or 51–300 lines                        | Full 5-layer review. All security phases, concise.                      |
| **DEEP**     | >10 files, >300 lines, or sensitive paths touched | Full 5-layer + extended security. Data flow tracing. Exploit scenarios. |

If any changed file matches sensitive patterns (`auth`, `crypto`, `token`, `session`, `password`, `.env`), the review always escalates to **DEEP**. Override with `--deep` or `--quick` flags.

### Multi-Layer Review

| Layer                 | Focus                                                                 |
| --------------------- | --------------------------------------------------------------------- |
| **0. Project Rules**  | CLAUDE.md compliance — MUST/SHOULD rule violations                    |
| **1. Plan Alignment** | Does the code match the intended design? Missing pieces?              |
| **2. Code Quality**   | Clarity, naming, consistency, complexity, performance, slop detection |
| **3. Security**       | Injection, auth bypasses, secrets, data flow — depth scaled by mode   |
| **4. Testing**        | Coverage gaps, edge cases, error paths, integration tests             |

All findings are confidence-scored (0–1). Only findings above 0.8 are reported, reducing false positives.

## Output

Findings from all 5 layers are collected and organized into 4 severity categories — no separate layer headings. Each finding is tagged with its source layer inline (e.g., `[L3]` for Security):

```
## 🔍 Code Review: src/auth/ (DEEP mode)

### 🚨 Critical Issues
- **[L3]** db/queries.ts:42 — Unsanitized SQL in query builder (confidence: 0.95)
  **[Injection]** — attacker can bypass auth via crafted email field
  → Use parameterized queries instead of string interpolation

### ⚠️ Important Issues
- **[L4]** api/auth.ts:18 — Missing rate limit test on login endpoint (confidence: 0.85)

### 💡 Suggestions
- **[L2]** src/auth/validators.ts:30 — Consider extracting validation logic into shared util

### 📋 Summary
Auth module is functional but has one critical SQL injection vulnerability in the query builder.
```

## External Reviewers

`/cf-review` can run headless external CLI reviewers in parallel with Claude's own multi-agent review. Each reviewer is invoked read-only, fed the same diff Claude sees, and its output is merged through the reducer with provenance tags (`[Codex]`, `[Gemini]`, `[Cursor]`, etc.).

| Flag       | CLI                             | Read-only mode                                                 |
| ---------- | ------------------------------- | -------------------------------------------------------------- |
| `--codex`  | `codex review` (dedicated path) | Codex's own review subcommand                                  |
| `--claude` | `claude -p`                     | `--permission-mode plan` (Claude's first-class read-only mode) |
| `--gemini` | `gemini -p`                     | `--approval-mode plan`¹                                        |
| `--cursor` | `cursor-agent -p`               | `--mode ask`¹                                                  |
| `--grok`   | `grok --prompt-file`            | `--sandbox read-only` (OS kernel sandbox — Seatbelt/Landlock)  |

¹ `gemini` and `cursor` use their documented read-only answer modes, but this has not been end-to-end verified. `claude` (plan mode) and `grok` (OS sandbox) are empirically verified to block repo writes while still returning a review. `grok`'s tool-permission flags (`--disallowed-tools`, `--permission-mode plan`) do **not** enforce read-only in headless mode — only the kernel `--sandbox` does.

Each agent runs under a per-agent timeout (default **300s**, override via `review.agentTimeout` in config). On timeout, the agent is dropped with a warning — the rest of the review still completes.

For a **manual round-trip** with any external AI (or a human), use `--out`: Claude's review runs first, then a prompt file is written with Claude's findings embedded as context. Collect results later via [`/cf-review-in`](/docs/skills/cf-review-in/).

## Dual Review with Codex

If you have the [Codex CLI](https://developers.openai.com/codex/cli/) installed and logged in, `/cf-review --with-codex` gets you a second independent opinion automatically — no copy-paste round-trip.

How it works:

1. The Codex review is spawned as a **background process**, so it runs concurrently with Claude's own multi-agent review — not one after the other. Codex auto-selects its scope from git state (`--base` for committed branch/unpushed work, `--uncommitted` for the working tree, `--commit` for a local-only repo), so it reviews a phase even after it has been committed.
2. Claude runs its normal 5-layer review in parallel.
3. When Codex finishes, its findings are normalized (`[P1]`→🚨 Critical, `[P2]`→⚠️ Important, `[P3]`→💡 Suggestion) and tagged `[Codex]`.
4. Both reviews are merged through the reducer — duplicate findings (same file:line) are collapsed, and agreement between Claude and Codex raises confidence.

Enable it by default for every review (including the auto-triggered ones from `/cf-plan`, `/cf-fix`, `/cf-optimize`) by setting `review.withCodex: true` in [`.coding-friend/config.json`](/docs/configuration/config-json/).

<Callout type="info" title="Graceful fallback">
  If any external reviewer is not installed, not logged in, times out, or
  errors, `/cf-review` prints a warning and continues without that reviewer — it
  never blocks. The built-in headless flags (`--claude`, `--gemini`, `--cursor`,
  `--grok`, `--codex`) are the no-copy-paste path.
  [`/cf-review-out`](/docs/skills/cf-review-out/) +
  [`/cf-review-in`](/docs/skills/cf-review-in/) remains the path for other
  external AIs (ChatGPT, Copilot) or human reviewers; [`/cf-review
  --out`](/docs/skills/cf-review/) bridges into that flow with Claude's context
  embedded.
</Callout>

## Auto-Triggered

`/cf-review` is automatically invoked by other skills — you don't always need to run it manually:

- [`/cf-plan`](/docs/skills/cf-plan/) — runs `/cf-review` after all implementation tasks complete
- [`/cf-fix`](/docs/skills/cf-fix/) — runs `/cf-review` after the fix is verified
- [`/cf-optimize`](/docs/skills/cf-optimize/) — runs `/cf-review` after the optimization is measured and verified

## When to Use

- **Before merging** — catch issues before they reach `main`
- **After implementing a feature** — verify quality before [`/cf-commit`](/docs/skills/cf-commit/)
- **Security-sensitive code** — auth, payments, data handling, crypto
- **Performance-critical paths** — database queries, hot loops, API endpoints
- **Unfamiliar code** — code you inherited or haven't touched in a while
- **Learning** — see what a thorough review looks like for mentorship

## Part of the Workflow

`/cf-review` fits naturally into the Coding Friend pipeline:

```
/cf-plan → implement (TDD) → (auto) /cf-review → /cf-ship
/cf-fix → (auto) /cf-review → /cf-commit
/cf-optimize → measure → optimize → (auto) /cf-review → /cf-commit
```

You can also use it standalone at any point. Combined with [`/cf-ship`](/docs/skills/cf-ship/) (which runs verification + commit + push + PR), it forms a complete quality gate before code reaches production.

## Defense in Depth

`/cf-review` is one layer in Coding Friend's [multi-layer security approach](/docs/reference/security/):

| Layer | Component                                                               | When            | What                                           |
| ----- | ----------------------------------------------------------------------- | --------------- | ---------------------------------------------- |
| 1     | [`privacy-block` hook](/docs/reference/security/#file-level-protection) | Every tool call | Blocks reading `.env`, secrets, keys           |
| 2     | [`/cf-commit`](/docs/skills/cf-commit/) secret scan                     | On commit       | Scans staged diff for leaked secrets           |
| 3     | `/cf-review` Layer 3                                                    | On review       | Deep security analysis with confidence scoring |

## Built-in Security Scan

After the 5-layer review, `/cf-review` automatically invokes Claude Code's built-in [`/security-review`](https://github.com/anthropics/claude-code-security-review) — a dedicated security audit skill from Anthropic. This means every `/cf-review` run gets **two layers of security analysis**:

| Pass                   | Source                                                                                   | Focus                                                                  |
| ---------------------- | ---------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| **Layer 3**            | Coding Friend's 5-layer review                                                           | Injection, auth bypasses, secrets, data flow — confidence-scored       |
| **`/security-review`** | [Anthropic's security-review](https://github.com/anthropics/claude-code-security-review) | Multi-phase vulnerability hunting with strict false-positive filtering |

Findings from both passes are **merged and deduplicated** — when both flag the same issue, the higher-severity entry is kept. This gives you comprehensive security coverage without running a separate command.
