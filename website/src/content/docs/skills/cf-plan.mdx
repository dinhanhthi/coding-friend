The `/cf-plan` skill helps you design solutions before writing code. It clarifies requirements, explores your codebase, brainstorms approaches, and produces a detailed implementation plan — then optionally implements it for you.

## Usage

```bash
/cf-plan [task description]
```

## Workflow

1. **Clarify Before Exploring** — Lists ambiguities and assumptions, asks you to confirm before proceeding. If the task involves external tools or libraries, asks which ones you prefer — no guessing
2. **Codebase Exploration** — Launches the [`cf-explorer`](/docs/reference/agents/#cf-explorer) agent to gather context: project structure, affected files, existing patterns, and relevant tests
3. **Brainstorm Approaches** — Launches the [`cf-planner`](/docs/reference/agents/#cf-planner) agent to generate 2–3 possible approaches, each with pros, cons, effort estimate, risk, and confidence level. Recommends one with rationale
4. **Validate with You** — Presents key findings, approaches, and open questions. Waits for your approval before writing anything
5. **Write & Save the Plan** — Breaks the approved approach into small, sequential tasks with specific files, expected outcomes, and verification steps. Saves to `docs/plans/`
6. **Offer Implementation** — Asks if you want to start implementing. If yes, executes each task sequentially using the [`cf-implementer`](/docs/reference/agents/#cf-implementer) agent with [TDD discipline](/docs/skills/cf-tdd/), then automatically runs [`/cf-review`](/docs/skills/cf-review/)

## Example

```bash
/cf-plan Build a REST API with JWT authentication and rate limiting
```

The skill will:

- Ask which auth library and rate limiting strategy you prefer
- Explore your existing API structure and auth setup
- Propose 2–3 approaches with trade-offs
- Present findings and wait for your approval
- Create a phased implementation plan
- Save the plan to `docs/plans/2026-03-05-rest-api-auth-plan.md`
- Offer to implement it task by task

## When to Use

- Starting a new feature with uncertain scope
- Redesigning a complex system
- Onboarding to unfamiliar codebases
- Planning migrations or refactors

## Output

Plans are saved to `docs/plans/` with a date-prefixed filename:

```
docs/plans/
  ├── 2026-03-05-rest-api-auth-plan.md
  ├── 2026-02-20-database-migration-plan.md
  └── ...
```

Each plan includes:

- Context and problem statement
- Confirmed assumptions with basis
- Chosen approach and rationale
- Sequential tasks with specific files and verification steps
- Risk assessment and mitigations
- **Next steps** — a reminder to run [`/cf-review`](/docs/skills/cf-review/) before committing

## Recommended Workflow

```
/cf-plan → implement (implicitly via /cf-implementer) → /cf-review → /cf-commit
```

After implementation, [`/cf-review`](/docs/skills/cf-review/) runs automatically — no need to trigger it manually before [`/cf-commit`](/docs/skills/cf-commit/).
