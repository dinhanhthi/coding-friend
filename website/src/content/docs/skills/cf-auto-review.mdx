The `cf-auto-review` skill is the **single source of truth** for Coding Friend's review methodology. It defines a two-pass review process (quick scan + deep 4-layer analysis), security vulnerability categories, confidence scoring, and severity levels used by all review components.

## Architecture

Three components work together for code review, each with a distinct role:

```
/cf-review (entry point — workflow orchestration)
    ↓ fork + dispatch
cf-code-reviewer agent (execution — how to review)
    ↓ loads
cf-auto-review (methodology — what to check)
```

| Component                                                      | Role                                                                   | Defined in                   |
| -------------------------------------------------------------- | ---------------------------------------------------------------------- | ---------------------------- |
| [`/cf-review`](/docs/skills/cf-review/)                        | Entry point. Gathers diff, detects review mode, dispatches to subagent | `cf-review/SKILL.md`         |
| [`cf-code-reviewer`](/docs/reference/agents/#cf-code-reviewer) | Execution. Reads code, applies methodology, formats report             | `agents/cf-code-reviewer.md` |
| `cf-auto-review`                                               | Methodology. Two-pass review (quick scan + 4-layer deep)               | `cf-auto-review/SKILL.md`    |

The `cf-code-reviewer` agent loads `cf-auto-review` at review time — it doesn't duplicate the methodology, it references it. This keeps the review methodology in one place.

**Without `/cf-review`**: When Claude reviews code casually (not via the slash command), it loads `cf-auto-review` directly inline — same methodology, no subagent.

## When It Activates

Automatically loaded when:

- The `cf-code-reviewer` agent runs a review
- Claude reviews code changes inline (without `/cf-review`)
- Assessing code quality before merge or commit

## Two-Pass Review

### Pass 1: Quick Scan

Before the deep review, a fast structural scan catches high-confidence issues:

1. **Obvious bugs** — null dereferences, unclosed resources, missing returns, off-by-one errors
2. **Structural issues** — missing imports, unused variables, broken type signatures
3. **Naming red flags** — single-letter variables in non-trivial code, misleading names
4. **Forgotten artifacts** — TODO/FIXME/HACK comments, console.log, debug code, commented-out code

Findings are reported immediately as **Critical** or **Important**.

### Pass 2: Deep 4-Layer Review

### Layer 1: Plan Alignment

Does the code implement what was planned?

- Unexpected changes outside the plan scope?
- Missing planned items?

### Layer 2: Code Quality

- **Naming** — variables, functions, and files named clearly?
- **Complexity** — unnecessary abstraction? Can functions be simplified?
- **Duplication** — repeated code that should be extracted?
- **Error handling** — errors handled at the right level? No swallowed errors?
- **Edge cases** — boundary conditions handled?

### Layer 3: Security

Depth scales by review mode (QUICK / STANDARD / DEEP):

**Phase 1: Context Research** (skipped in QUICK mode)

- Identify existing security frameworks/libraries
- Look for established sanitization/validation patterns
- Understand trust boundaries

**Phase 2: Vulnerability Assessment**

| Category             | Look for                                                                  |
| -------------------- | ------------------------------------------------------------------------- |
| **Input Validation** | SQL injection, command injection, XXE, template injection, path traversal |
| **Auth & Access**    | Auth bypass, privilege escalation, session flaws, JWT issues              |
| **Secrets & Crypto** | Hardcoded keys/tokens, weak crypto, improper key storage                  |
| **Code Execution**   | RCE via deserialization, eval injection, XSS (reflected/stored/DOM)       |
| **Data Exposure**    | Sensitive data in logs, PII handling, API endpoint leakage, debug info    |
| **Prompt Injection** | External content targeting AI without sanitization                        |

**Method**: Trace data flow from user inputs → through processing → to sensitive operations. Flag where untrusted data crosses trust boundaries without validation.

**Phase 3: Confidence Filtering**

Score each finding 0.0–1.0. Only report findings at **0.8 or above** — anything lower is too speculative.

**False positives** (do NOT flag):

- UUIDs as identifiers, environment variables / CLI flags
- Framework default protections (React auto-escaping, etc.) unless explicitly bypassed
- Client-side permission checks, logging non-PII data, DoS / rate limiting

### Layer 4: Testing

- **Coverage** — new code paths tested?
- **Quality** — tests verify behavior, not implementation?
- **Edge cases** — error paths and boundary conditions tested?
- **Regression** — would tests catch the bug if reintroduced?

## Severity Levels

| Level          | Meaning                                   | Action                |
| -------------- | ----------------------------------------- | --------------------- |
| **Critical**   | Bug, security hole, data loss risk        | Must fix before merge |
| **Important**  | Design issue, missing test, poor naming   | Should fix            |
| **Suggestion** | Style, alternative approach, nice-to-have | Consider              |
